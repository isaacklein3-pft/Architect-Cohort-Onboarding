Section 01

Purpose and Scope

This document is M4's entry gate. No operator joins the live Superstructure founding cohort without clearing every required check in this audit. The specifications defined in M2 and M3 describe what the system should do — this audit verifies that each specific operator can actually do it before real institutional assets are involved.

The audit covers four domains: live telemetry and OCP integration verification, signing ceremony dry-run testing, administrative onboarding (escalation contacts, SLA acknowledgment, custody attestations), and the remediation workflow for operators who fail initial checks. Each domain maps to one of the four task steps below. Every check is either a pass or a fail — there are no partial scores or weighted averages. An operator who passes 14 of 15 checks has not cleared the audit.

The audit is a one-time gate before cohort activation, not an ongoing monitoring framework. Ongoing monitoring is handled by the OCP telemetry system defined in M3. This document covers the period between an operator completing their technical setup and the moment their validator goes live with institutional participation.

Design Decision — Binary Pass/Fail vs Weighted Scoring A weighted scoring model would let an operator compensate for a weak area (say, slightly high ceremony latency) with strength in another (excellent uptime history). We rejected this because the M3 specs define specific thresholds for operational reasons — the 2,000ms dry-run latency ceiling isn't a preference, it's the boundary above which the 120-second freshness window becomes a real operational risk. Allowing compensating factors would introduce negotiation into what should be a binary technical verification. The cost is that operators with one genuine gap cannot advance until it is fixed. That cost is the point.
Section 02

Assumption Registry and Open Questions

ASSUMPTION-01 Confirmed

Audit checks are performed by Superstructure operations personnel, not by automated systems alone. Automated checks flag pass/fail against defined thresholds. A human reviewer from Superstructure operations signs off on each operator's full audit record before cohort activation is granted. Automated pass on every check is necessary but not sufficient.

ASSUMPTION-02 Confirmed

All M3 thresholds apply directly to audit criteria. The OCP telemetry polling interval (30 seconds), CRS recalculation interval (60 seconds), failover trigger thresholds, and signing ceremony freshness window (120 seconds) are carried forward unchanged. This audit verifies compliance with those thresholds — it does not redefine them.

ASSUMPTION-03 Confirmed

Signing ceremony dry runs use test envelopes with a designated test actionClass. Test envelopes are structurally identical to production envelopes — same schema, same field requirements, same freshness window — but are flagged as test payloads so they do not trigger any on-chain action. The Maker/Checker/Reviewer sequence must be completed in full even for test envelopes.

ASSUMPTION-04 Open

Dry run test envelopes are issued by Superstructure operations. The mechanism for issuing test envelopes — whether through a dedicated test endpoint in the middleware or through a manual envelope construction process — has not been confirmed by Post Fiat core. This audit spec treats test envelope issuance as an operations responsibility pending that confirmation.

UNKNOWN-01 — Verified Inference Score baseline for new operators

A newly onboarded operator has no Verified Inference Score history. The audit uses uptime rate and telemetry delivery as the baseline proxy. The question of whether Post Fiat assigns a default starting VI score or leaves it at zero until first scored period is not yet confirmed. If zero, new operators would technically fail CRS calculation on first check. This spec defines the audit baseline as telemetry delivery rate rather than CRS-derived VI score for that reason.

UNKNOWN-02 — Minimum soak period for telemetry baseline

This spec defines a 72-hour telemetry soak period before baseline validation. Whether Post Fiat's OCP requires a different minimum observation window before an operator is eligible for CRS scoring is not confirmed.

Section 03

Audit Gate Model

The audit runs in four sequential gates. An operator cannot advance to Gate 2 until Gate 1 is cleared. This sequencing is deliberate — there is no point running a signing ceremony dry run against an operator whose telemetry infrastructure is not live, and no point completing administrative onboarding for an operator who cannot pass a dry run.

Gate 1
Telemetry
Live verification · OCP ingestion · CRS baseline
Gate 2
Ceremony
Dry run · Latency · Role sequence · Evidence
Gate 3
Admin
Escalation contacts · SLA · Custody attestation
Final Gate — Superstructure Operations Sign-Off

Cohort Activation Authorized

All three gates cleared. Superstructure operations reviewer signs the operator audit record. Middleware fires OPERATOR_AUDIT_COMPLETE. Operator enters READY state. Institutional deployment can proceed per the M3 Institutional Deployment Pathway.

Severity Classification for Checklist Items

Critical — Fail blocks gate advancement. Must be resolved before any further audit activity.
High — Fail blocks gate advancement. 48-hour remediation window before retest.
Standard — Fail triggers 7-day remediation window. Gate advancement blocked until resolved.

Middleware confirms operator telemetry payloads are being received at the OCP ingestion endpoint. Zero payloads received after 72-hour soak = immediate fail. Evidence: OCP ingestion log showing operator telemetry records with timestamps.

Telemetry delivery rate ≥ 98% over 72-hour soak
Of all expected 30-second poll intervals during the soak period, at least 98% must have delivered payloads. A 72-hour window contains 8,640 expected polls — at least 8,468 must have been received. Evidence: delivery rate calculation log from OCP.
T-03 · Critical
No consecutive missed poll sequence exceeding 3 during soak
Any sequence of 4 or more consecutive missed polls during the soak period is an automatic fail, regardless of overall delivery rate. This mirrors the production ERR_TELEMETRY_GAP trigger and confirms the operator's telemetry system does not suffer from periodic outage patterns. Evidence: consecutive gap analysis from OCP ingestion log.
T-04 · High
All telemetry payloads pass schema validation
Every received payload must contain all required fields typed correctly per the M3 Operator Telemetry Payload schema: operatorId, validatorPubKey, timestamp, consensusParticipationStatus, uptimeRate24h, latencyP95Ms, verifiedInferenceScore, ledgerParticipationRate, allocationWeightCurrent, maintenanceDeclared, telemetrySignature. Any payload with missing or malformed fields is a schema validation failure. Evidence: schema validation pass rate log.
T-05 · High
All telemetry payloads carry valid Ed25519 telemetrySignature
The telemetrySignature field must be present and valid on every payload. Unsigned or invalid-signature payloads cannot be trusted by the OCP in production and confirm the operator has not completed their signing infrastructure setup. Evidence: signature validation pass rate from OCP.
T-06 · High
Uptime rate ≥ 99.0% over 72-hour soak period
The uptimeRate24h field averaged across all delivered payloads during the soak must be at or above 99.0%. This matches the M3 production SLA floor. Evidence: uptimeRate24h average from OCP payload records.
T-07 · High
p95 latency ≤ 350ms over 72-hour soak period
The p95 latency value from latencyP95Ms field must stay at or below 350ms throughout the soak period — 10% tighter than the 400ms production failover trigger. Operators entering the cohort at exactly the failover threshold have no operational headroom. Evidence: latencyP95Ms distribution from OCP payload records.
T-08 · Standard
CRS baseline ≥ 75 on first completed 60-second scoring cycle
After the soak period, the OCP runs its first full CRS calculation against the operator. The resulting score must be at or above 75. This confirms the full scoring pipeline — telemetry ingestion, field parsing, CRS calculation — is functioning end-to-end, and that the operator is above the 70 degraded threshold with headroom. Evidence: first CRS calculation output from OCP with component scores.
T-09 · Standard
OCP routing weight assignment confirmed
After CRS baseline is established, the OCP must successfully assign an initial routing weight to the operator within one scoring cycle (60 seconds). Weight must be within the policy bounds defined in the institution's allocation policy (default: 10%–60%). Evidence: routing weight assignment log from OCP with policy hash and CRS snapshot.
Step 02 — Gate 2

Signing Ceremony Dry-Run Pass/Fail Conditions

Gate 2 confirms the operator's custody environment can support the signing ceremony workflow under production-equivalent conditions. The dry run uses test envelopes — structurally identical to production envelopes but flagged as test payloads. Every requirement that applies in production applies in the dry run. No exceptions.

The operator must complete two dry runs: one with a routine test payload (simulating a standard governance vote) and one with a time-pressure test payload delivered with a 90-second notice (simulating an emergency signing scenario). Both must pass independently.

Gate 2 Checklist

C-01 · Critical
Envelope delivery latency ≤ 2,000ms at p95
Superstructure operations delivers the test envelope and timestamps issuance. The operator's custody environment must confirm receipt within 2,000ms at p95 across the two dry runs. Latency above this ceiling means the freshness window becomes a real operational risk in production, particularly during emergency scenarios. Evidence: envelope delivery timestamp log from Superstructure ops + operator custody receipt confirmation.
C-02 · Critical
Full Maker/Checker/Reviewer sequence completed with three separate individuals
The dry run must be completed with three distinct individuals occupying the Maker, Checker, and Reviewer roles. The operator must provide role attestation records identifying each individual by role. Any attempt to have one person occupy two roles is an immediate fail — this is the institutional control that M2 and M3 define as non-negotiable. Evidence: role attestation record with three distinct role holders.
C-03 · Critical
Reviewer role not waived
Separate from the three-person check — the Reviewer role must be explicitly present in the ceremony record. An operator who completes Maker and Checker but cannot staff a Reviewer has not passed the dry run, regardless of how quickly the two-step sequence was completed. Evidence: ceremony completion record with Reviewer authorization field populated.
C-04 · Critical
Checker independently verified payload hash before signing
The Checker must provide a written attestation confirming they independently verified the test payload hash against the test specification document before signing — not relying on the Maker's confirmation. This is the explicit control that prevents a substituted or malformed payload from being signed under time pressure. Evidence: Checker verification attestation with hash comparison record.
C-05 · High
Routine dry run completed within 300 seconds of envelope receipt
The full Maker/Checker/Reviewer sequence for the routine dry run must complete within 300 seconds of envelope receipt. This tests that the operator's ceremony workflow is rehearsed, not improvised. Evidence: ceremony completion timestamp vs envelope receipt timestamp.
C-06 · High
Emergency dry run completed within 180 seconds of envelope receipt
The emergency dry run, delivered with 90-second notice, must complete within 180 seconds of envelope receipt. This confirms the operator can mobilize all three roles quickly under emergency conditions. An operator who passes the routine dry run but fails the emergency dry run has not demonstrated production readiness for Severity 1 events. Evidence: ceremony completion timestamp vs envelope receipt timestamp for emergency run.
C-07 · High
Signed payload hash matches test envelope canonical payload
The signed payload returned by the operator must have a verifiable hash match to the canonical payload in the test envelope. A hash mismatch means the operator signed something other than what was delivered — the most serious possible ceremony failure. Evidence: Superstructure ops hash verification of returned signed payload.
C-08 · High
Replay protection produces ERR_NONCE_REPLAY on duplicate nonce resubmission
After a successful dry run, Superstructure ops resubmits the same ceremony with the same nonce. The operator's custody environment must reject the resubmission with ERR_NONCE_REPLAY. Failure to reject means the operator's nonce log is not functioning — a replay attack would succeed in production. Evidence: resubmission rejection log showing ERR_NONCE_REPLAY.
C-09 · Standard
Nonce log retention confirmed at 30-day minimum
Operator must confirm in writing that their nonce retention period is at least 30 days and provide the storage mechanism used (database, HSM-side log, or equivalent). Evidence: written nonce retention confirmation with mechanism description.
C-10 · Standard
Custody environment type confirmed as approved
Operator must confirm their custody environment is one of the three approved types from M2: institution-owned HSM (FIPS 140-2 Level 3 or above), air-gapped signing system, or institutional multisig with institution as policy owner. Cloud HSM configurations where the provider retains operational access are not approved. Evidence: written custody environment attestation with hardware/software specification.
Step 03 — Gate 3

Administrative Onboarding Requirements

Gate 3 covers the administrative and compliance requirements that sit alongside the technical checks. These are not formalities — an operator who has not registered escalation contacts cannot be reached during a Severity 1 incident, and an operator who has not acknowledged the SLA has not accepted the operational obligations that the cohort depends on.

Gate 3 Checklist

A-01 · Critical
Primary and secondary escalation contacts registered with Superstructure operations
Operator must register a minimum of two escalation contacts: a primary (technical lead, 15-minute response SLA for Severity 1) and a secondary (backup, activated if primary is unresponsive after 15 minutes). Each contact must include: full name, role, direct phone number, email, and timezone. Contacts must be individuals, not shared inboxes or on-call rotation aliases. Evidence: escalation contact registration form with all fields completed, confirmed by Superstructure ops.
A-02 · Critical
Escalation contact reachability confirmed via test contact
Superstructure operations must have successfully reached both the primary and secondary contacts via direct phone within 30 days before audit completion. A contact that cannot be reached is not a valid escalation contact regardless of registration. Evidence: test contact log from Superstructure ops showing successful reach of both contacts.
A-03 · Critical
SLA acknowledgment signed and on file
Operator must provide a signed SLA acknowledgment confirming acceptance of the operational obligations defined in the M3 Validator Operator Integration Specification: 99.0% minimum uptime, 400ms p95 latency ceiling, 30-second telemetry polling interval, 15-minute Severity 1 response, 4-hour RCA acknowledgment window. The SLA acknowledgment must be signed by an authorized representative of the operator organization, not a technical contact. Evidence: signed SLA acknowledgment document on file with Superstructure operations.
A-04 · High
Key custody attestation submitted and reviewed
Operator must submit a written attestation confirming: the validator key is held in an approved custody environment; the key has never been held in a cloud HSM where the provider retains operational access; the key has never been transmitted in plaintext; and the custody environment is operated exclusively by the operator organization with no third-party operational access. Evidence: signed key custody attestation reviewed by Superstructure operations.
A-05 · High
Insurance confirmation active per M2 Insurance Architecture
Operator must confirm that their insurance coverage satisfying the M2 Insurance and Indemnification Architecture requirements is in force. Coverage must not expire within 90 days of cohort activation date. Evidence: insurance confirmation letter or certificate of coverage with expiration date reviewed by Superstructure operations.
A-06 · High
Geographic and infrastructure diversity requirements met
Operator's infrastructure must not create concentration violations: no more than 2 operators per jurisdiction, no more than 2 operators per cloud provider or colocation facility within the founding cohort. Superstructure operations maintains the cohort diversity map and verifies the operator's proposed infrastructure against it before this check passes. Evidence: Superstructure ops diversity check against current cohort map.
A-07 · Standard
Governance forum access confirmed
Operator has created an account on the governance forum and has posted at minimum one introductory message in the founding cohort channel. This confirms they can access the primary deliberation venue before any governance events occur. Evidence: governance forum account with verifiable post in founding cohort channel.
A-08 · Standard
Operational communication channel membership confirmed
Operator's primary escalation contact is a member of the Tier 2 operational coordination channel and has acknowledged the channel's response time expectations (4 hours for operational alerts, 15 minutes for degraded validator alerts). Evidence: channel membership confirmation from Superstructure ops.
Step 04

Remediation Workflow and Grace Period Definitions

Every failed check has a defined remediation path and a retest window. There are no open-ended remediation periods — an operator who has not resolved a failed check within its window either receives one extension (with Superstructure operations review) or is deferred from the founding cohort.

Remediation Tracks by Severity

Critical Failure — Immediate Block 48-hour remediation window

A Critical check failure blocks all further audit activity immediately. The operator must:

  • Identify the specific failure mode within 4 hours of notification
  • Submit a written root cause description and remediation plan to Superstructure operations within 12 hours
  • Resolve the failure and request retest within 48 hours of the original failure

If the 48-hour window is missed, the operator may request a single 48-hour extension with written justification. Extensions require Superstructure operations approval and are granted once per check per audit cycle. A second failure on the same Critical check after the extension window expires results in cohort deferral — the operator exits the current audit cycle and must restart the full audit process after a 30-day waiting period.

High Failure — Gate Block 7-day remediation window

A High check failure blocks gate advancement but does not halt audit activity within the current gate. The operator must:

  • Submit a root cause description within 48 hours of notification
  • Resolve the failure and request retest within 7 days of the original failure

If the 7-day window is missed, the operator may request a single 7-day extension. Extensions require Superstructure operations approval. A second failure on the same High check after the extension window expires results in cohort deferral with a 14-day waiting period before restart.

Standard Failure — Tracked Resolution 14-day remediation window

A Standard check failure is logged and tracked but does not block concurrent audit activity within the same gate. The operator must:

  • Acknowledge the failure within 48 hours
  • Resolve and request retest within 14 days

Standard failures cannot be carried past gate completion — an operator cannot advance to the next gate with any unresolved Standard failures. Two unresolved Standard failures at the point of gate advancement attempt trigger a Superstructure operations review before any extension is granted.

Remediation Workflow — Step by Step

  • 1

    Failure Notification

    Superstructure operations notifies the operator of the failed check within 2 hours of audit finding. Notification includes: check ID, failure description, specific threshold or requirement that was not met, remediation window start time, and window expiry timestamp.

    Event: OPERATOR_AUDIT_CHECK_FAILED
  • 2

    Root Cause Submission

    Operator submits written root cause description within the required window (4 hours for Critical, 48 hours for High and Standard). Root cause must identify: the specific technical or operational failure that caused the check to fail, what configuration or process change is being made to resolve it, and confirmation that the fix will not introduce new failure modes.

    Event: OPERATOR_REMEDIATION_PLAN_RECEIVED
    Fail: ERR_REMEDIATION_PLAN_NOT_SUBMITTED — no response within required window
    Rollback: Critical — automatic cohort deferral; High/Standard — escalate to Superstructure operations review
  • 3

    Retest Request

    Operator notifies Superstructure operations that the failure has been resolved and requests retest. Superstructure operations schedules retest within 48 hours of request for Critical and High failures, within 5 business days for Standard failures.

    Event: OPERATOR_RETEST_REQUESTED
  • 4

    Retest Execution

    The specific failed check is re-run against the same pass/fail criteria. A passing retest clears the check and resumes gate progression. A failing retest starts the extension evaluation: was the original remediation window fully used? Was the root cause accurately identified? Superstructure operations makes the extension/deferral decision within 24 hours of retest failure.

    Event: OPERATOR_AUDIT_CHECK_PASSED
    Event: OPERATOR_AUDIT_DEFERRED — second consecutive fail after extension
  • 5

    Gate Completion or Deferral

    Once all checks in a gate are cleared, Superstructure operations marks the gate complete and logs OPERATOR_GATE_COMPLETE. The operator advances to the next gate. If any check results in cohort deferral, Superstructure operations issues a deferral notice with the waiting period duration and the conditions required before the operator may restart the audit.

    Event: OPERATOR_GATE_COMPLETE
    Event: OPERATOR_AUDIT_DEFERRED

Section 09

Error Codes and Typed Failure Reasons

ERR_TELEMETRY_DELIVERY_RATE_BELOW_THRESHOLD
Gate 1 — T-02 Fail

Telemetry delivery rate fell below 98% during 72-hour soak. OCP delivery rate calculation logged with exact percentage. Critical severity.

ERR_CONSECUTIVE_POLL_GAP_EXCEEDED
Gate 1 — T-03 Fail

Four or more consecutive missed polls detected during soak period. Gap start and end timestamps logged. Critical severity.

ERR_TELEMETRY_SCHEMA_VALIDATION_FAILURE
Gate 1 — T-04 Fail

One or more telemetry payloads failed schema validation. Missing or malformed fields logged per payload. High severity.

ERR_TELEMETRY_SIGNATURE_INVALID
Gate 1 — T-05 Fail

One or more payloads had missing or invalid telemetrySignature field. High severity.

ERR_CRS_BASELINE_BELOW_THRESHOLD
Gate 1 — T-08 Fail

First CRS calculation returned score below 75. Component scores logged for diagnosis. Standard severity.

ERR_CEREMONY_ENVELOPE_LATENCY_EXCEEDED
Gate 2 — C-01 Fail

Envelope delivery latency exceeded 2,000ms at p95. Delivery timestamp log included. Critical severity.

ERR_CEREMONY_ROLE_COLLAPSE
Gate 2 — C-02 Fail

Same individual occupied two or more roles in the ceremony sequence. Immediate fail. Critical severity.

ERR_REVIEWER_ABSENT
Gate 2 — C-03 Fail

Reviewer role not present in ceremony completion record. Ceremony completed with Maker and Checker only. Critical severity.

ERR_CHECKER_VERIFICATION_NOT_ATTESTED
Gate 2 — C-04 Fail

Checker did not provide independent hash verification attestation. Critical severity.

ERR_DRY_RUN_TIMEOUT
Gate 2 — C-05 or C-06 Fail

Ceremony completion time exceeded threshold (300s routine, 180s emergency). Completion timestamp logged. High severity.

ERR_SIGNED_PAYLOAD_HASH_MISMATCH
Gate 2 — C-07 Fail

Returned signed payload hash does not match canonical payload from test envelope. High severity. Requires investigation before retest.

ERR_REPLAY_PROTECTION_NOT_FUNCTIONING
Gate 2 — C-08 Fail

Resubmission of used nonce was accepted rather than rejected with ERR_NONCE_REPLAY. High severity.

ERR_ESCALATION_CONTACT_UNREACHABLE
Gate 3 — A-02 Fail

Primary or secondary escalation contact could not be reached via direct phone during test contact. High severity.

ERR_REMEDIATION_PLAN_NOT_SUBMITTED
Remediation — Step 2 Fail

Operator did not submit root cause description within required window. Triggers escalation to Superstructure operations review. Severity depends on original check.

{

Section 11

  • OPERATOR_AUDIT_INITIATED Audit cycle begins; 72-hour telemetry soak period starts; OCP ingestion monitoring active

  • OPERATOR_GATE_COMPLETE All checks in a gate cleared; operator advances to next gate
  • OPERATOR_GATE_2_INITIATED Gate 1 complete; dry run test envelopes issued; routine and emergency runs scheduled

Runbook — Full Audit Cycle

  1. Initiate audit — log OPERATOR_AUDIT_INITIATED, confirm OCP ingestion active, start 72-hour soak clock
  2. After 72 hours: run Gate 1 checks T-01 through T-09 against soak data; log each result
  3. For any failed checks: issue failure notification within 2 hours; track remediation window
  4. Once all Gate 1 checks pass: log OPERATOR_GATE_COMPLETE, initiate Gate 2 — schedule routine dry run, set emergency dry run date (unannounced until 90-second notice)
  5. Run routine dry run; run checks C-01 through C-10; log results
  6. Run emergency dry run on separate day with 90-second notice; re-run timing checks C-05/C-06 under emergency conditions
  7. Run replay protection test C-08 after any successful dry run
  8. Once all Gate 2 checks pass: log OPERATOR_GATE_COMPLETE, initiate Gate 3 — issue admin documentation requests
  9. Run Gate 3 checks A-01 through A-08 as documents are received; log results
  10. Once all Gate 3 checks pass: Superstructure ops reviewer reviews full audit record, signs off, logs OPERATOR_AUDIT_COMPLETE
  11. Operator enters READY state; institutional deployment proceeds per M3 Institutional Deployment Pathway

Section 14

Zero-Custody Audit Assurance

Cohort Onboarding Custody Guarantee

No part of this audit process requires an operator to transmit key material, expose their custody environment to Superstructure operations, or demonstrate key custody through any mechanism other than signed attestation and dry run ceremony completion.

The signing ceremony dry runs verify that the operator's custody environment can complete the ceremony workflow — they do not expose the key. Test envelopes are flagged as test payloads and produce no on-chain action. The signed test payload hash confirms the ceremony completed correctly. The key that signed it remains in the operator's custody environment throughout.

The audit gate model confirms operational readiness. It does not transfer any custody rights to Superstructure operations. An operator who completes this audit and enters READY state has demonstrated they can participate in the cohort under the zero-custody model — they have not granted any party access to their key as part of that demonstration.
